    • Introdução

    Apesar de ter sido desenvolvido no âmbito da unidade curricular de LPROG, o trabalho apresentado enquadra-se no domínio do Projeto Integrador do Semestre 4 na medida em que os ficheiros gerados (XML) representam a informação que se encontra registada no Sistema.
    Os principais objetivos foram:

  • Definir uma representação XML para um domínio específico.
  • Definir a sintaxe dessa linguagem através do respetivo XML Schema (XSD).
  • Criar instâncias (documentos XML) e validá-las de acordo com o XSD.
  • Fazer transformações ao documento XML (para JSON, TXT e HTML) usando XSLT e XPath.

    • Análise

    O ficheiro XML é gerado quando é selecionada a opção de exportação no Sistema, assim o sistema disponibiliza toda a informação que possui a sistemas externos.
    Visto que a configuração do documento XML é escolhida pelo utilizador (na medida em que este é quem decide quais conceitos pretende exportar de uma lista de conceitos fornecidos), a grande maioria dos conceitos são opcionais.
    A equipa decidiu criar uma estrutura hierárquica que parte de um elemento geral que se divide em vários outros elementos gerais o ficheiro em Areas Funcionais: Chão de Fábrica e Produção.
    A Área da Produção é responsável pela informação relativa a ordens de produção, produtos, matérias-primas, entre outros. Já a área do Chão de Fábrica é responsável pelas linhas de produção e respetivas máquinas e depósitos.
    Dentro de cada Àrea Funcional foram identificados os diversos conceitos que fossem possíveis de exportar assim como sub-conceitos a eles associados. Assim, podemos distinguir os seguintes conceitos-chave:

  • Matérias Primas
  • Categorias de Matérias Primas
  • Produtos
  • Ordens de Produção
  • Relatórios de Ordens de Produção
  • Linhas de Produção
  • Máquinas
  • Depósitos

    XSD / XML Schema

    O ficheiro XSD foi utilizado para definir quais os elementos e atributos podem aparecer no documento XML. Considerando a utilização do XSD foi definido que cada atributo e cada elemento, complexo ou simples, foram definidos utilizando diferentes types e quando esses elementos necessitavam de ser usados eram apenas referenciados. Deste modo o ficheiro XSD torna-se muito mais simples de manter e reutilizar.
    Por exemplo, para o ficheiro raiz do XSD, temos a seguinte definição:

    Código Exemplo: 
    
                        xs:element name="fabrica"
                            xs:complexType
                                 xs:sequence
                                    xs:element name="producao" type="TProducao" minOccurs="0" maxOccurs="unbounded"/
                                    xs:element name="chaoDeFabrica" type="TChaoFabrica" minOccurs="0" maxOccurs="unbounded"/
                                /xs:sequence
                            /xs:complexType
                        /xs:element

    Onde o type TProducao e TChaoFabrica são definidos da seguinte forma, respectivamente:

    Código Exemplo: 
    
                        xs:complexType name="TProducao"
                            xs:sequence
                                xs:element name="RawMaterials" type="TCMateriasPrimas" minOccurs="0"/
                                xs:element name="RawMaterialCategorys" type="TCCategoriasMateriaPrima" minOccurs="0"/
                                xs:element name="Products" type="TCProdutos" minOccurs="0"/
                                xs:element name="ProductionOrders" type="TCOrdensProducao" minOccurs="0"/
                            /xs:sequence
                        /xs:complexType

    Código Exemplo: 
    
                        xs:complexType name="TChaoFabrica"
                            xs:sequence
                                xs:element name="ProductionLines" type="TCLinhasProducao" minOccurs="0"/
                                xs:element name="Deposits" type="TCDepositos" minOccurs="0"/
                            /xs:sequence
                        /xs:complexType

    Também foram utilizados Simple Types para identificar tipos que fossem necessários utilizar mais do que uma vez ou que necessitassem de validações como o exemplo abaixo exposto.

    Código Exemplo: 
    
                        xs:simpleType name="TFullDescription"
                            xs:restriction base="xs:string"
                                xs:maxLength value="70"/
                                xs:pattern value="[A-Za-z0-9 \wÀ-ú\.\-]+"/
                            /xs:restriction
                        /xs:simpleType

    Código Exemplo: 
    
                        xs:simpleType name="TUnidade"
                            xs:restriction base="xs:string"
                                xs:enumeration value="UN"/
                                xs:enumeration value="TONNE"/
                                xs:enumeration value="KILOGRAMS"/
                                xs:enumeration value="LITERS"/
                            /xs:restriction
                        /xs:simpleType
